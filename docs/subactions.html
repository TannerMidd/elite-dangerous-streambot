Sub-Actions & Variables Reference

Every value the standalone Elite Dangerous journal watcher puts at your fingertips inside Streamer.bot — trigger arguments, live global variables, and how to use them in any sub-action.

Prefer a guided build?

Open the Streamer.bot Action Builder, choose what should happen in Elite Dangerous, combine checks with AND or OR, add ordered Else-if and Else paths, and get the exact editable action tree, variables, result sub-actions, and testing steps.

The two kinds of values

The watcher hands data to your sub-actions through two different channels. Knowing which is which is the whole game:

Trigger argumentsGlobal variables
Syntax %edEvent_StarSystem% ~edSystem~
What they are A snapshot of the event that fired the trigger Your live game state, always current
Available Only inside an action started by a watcher trigger In any action, from any trigger (chat command, hotkey, timer…)
Where to browse them Action Queues → Action History → right-click a run → Inspect Variables Streamer.bot → (x) Global Variables → Persisted Globals

Both work in almost every sub-action text input — If/Else conditions, TTS text, chat messages, file writers, C# code. Type the token exactly, including the %…% or ~…~ wrapper.

The #1 mistake

Typing edSystem (no wrapper) into an If/Else Input compares the literal text "edSystem", which never matches anything. Use %edSystem% when the value came with the trigger, or ~edSystem~ to read the live global from anywhere. All the watcher's globals are persisted, so the tilde syntax always works for them.

Finding a field name when you're not sure

Journal events carry whatever fields the game writes, so this page can't list every field of every event — but you never have to guess:

  1. Inspect a real run (best). Let the trigger fire once (or play a minute with any action bound to Any Journal Event), then open Action Queues → Action History, right-click the run, and choose Inspect Variables. Streamer.bot shows the complete argument stack — every %edEvent_…% name next to its actual value, ready to copy into your sub-actions.
  2. Look at the event mirror. Open (x) Global Variables → Persisted Globals and filter for edEvt. After an event has happened once, every one of its fields sits there by name — e.g. seeing edEvtDocked_StationName tells you the trigger argument is %edEvent_StationName%. Works even for events that fired hours ago.
  3. Log the raw event. Bind a throwaway action to Any Journal Event with a single Core → Logging → Log Information sub-action whose message is %edEventName%: %edEventJson% — every field name is right there in the JSON.
  4. Check Frontier's journal manual. The community-hosted journal documentation lists each event's fields; prefix any field with edEvent_ to get the argument name.
Test without playing

You don't need to be in-game to populate Action History — right-click your action → Test Action runs it (without trigger arguments), and any past real run stays inspectable. For a realistic argument stack, the easiest source is simply the last time the trigger fired.

Trigger arguments — journal event triggers

Fired by Elite Dangerous → Any Journal Event and every Elite Dangerous → Journal Events → <Name> trigger. Both deliver the same arguments:

ArgumentHolds
%edEventName%the event type, e.g. FSDJump
%edEventTimestamp%the event's in-game UTC timestamp
%edEventJson%the complete raw event as JSON (capped at 4,000 chars)
%edEvent_<Field>%every field of the event, one argument each

Worked examples

TriggerTypical arguments you'd use
FSDJump%edEvent_StarSystem%, %edEvent_JumpDist%, %edEvent_FuelUsed%
Docked%edEvent_StationName%, %edEvent_StationType%, %edEvent_StarSystem%
Bounty%edEvent_TotalReward%, %edEvent_Target_Localised%, %edEvent_VictimFaction%
MissionCompleted%edEvent_Reward%, %edEvent_LocalisedName%, %edEvent_DestinationSystem%

Nested fields

Fields inside sub-objects join with underscores: FSDJump's SystemFaction.Name arrives as %edEvent_SystemFaction_Name%. Arrays arrive as a count plus JSON — see flattening rules.

Trigger arguments — Ship State triggers

Fired by every Elite Dangerous → Ship State → <Flag> On/Off trigger and by Any Ship Flag Changed:

ArgumentHolds
%edFlag%the flag that changed, e.g. LandingGearDown
%edValue%True or False — the flag's new state
%edSystem%the star system you're in when it changed

The full list of flags (each one is both a pair of triggers and a live global variable) is under Ship flags and On-foot flags below.

Trigger arguments — Companion File Updated

Fired when the game rewrites one of its side files (Cargo.json, NavRoute.json, Market.json, Backpack.json, ShipLocker.json, ModulesInfo.json, …):

ArgumentHolds
%edCompanionName%which file, e.g. Cargo
%edCompanionFile%the file name, e.g. Cargo.json
%edCompanionJson%the file's full contents as JSON (capped at 4,000 chars)

Trigger arguments — Watcher Status

Fired when the watcher itself changes state — useful for a startup confirmation sound or an error alert:

ArgumentHolds
%edStatus%one of started · stopped · hydrated · journal_changed · error
%edMessage%a human-readable detail line
%edJournalFile%the journal file currently being watched

Globals — Status.json extras

Live cockpit state beyond the flags, refreshed about once a second while the game runs:

VariableHolds
~edPipsSys~ / ~edPipsEng~ / ~edPipsWep~power pips (0–4, halves allowed)
~edFireGroup~selected fire group (0-based)
~edGuiFocus~ / ~edGuiFocusName~focused panel — number and name: NoFocus, InternalPanel, ExternalPanel, CommsPanel, RolePanel, StationServices, GalaxyMap, SystemMap, Orrery, FSS, SAA, Codex
~edFuelLevel~ / ~edFuelReservoir~main tank / reservoir, in tons

~edLegalState~Clean, Wanted, Speeding, …
~edBalance~credit balance
~edLatitude~ / ~edLongitude~ / ~edHeading~ / ~edAltitude~position near a body
~edBodyName~ / ~edPlanetRadius~body being approached / landed on
~edOxygen~ / ~edHealth~ / ~edTemperature~ / ~edGravity~on-foot vitals (0–1 for oxygen/health)

~edDestinationName~ / ~edDestinationSystem~ / ~edDestinationBody~locked destination
Position & vitals only exist sometimes

Latitude/longitude appear only near a body, and oxygen/health/temperature only on foot — the game omits them otherwise, so the variables simply keep their last value. Gate on ~edHasLatLong~ or ~edOnFoot~ if it matters.

Globals — navigation

VariableHolds
~edNavTargetSystem~next system on your plotted route
~edNavDestination~final system of the route
~edNavRemainingJumps~jumps left to the destination

Globals — event mirrors

Every journal event is also mirrored into globals, so the latest occurrence of anything is always readable — even from actions that weren't triggered by it:

PatternExampleHolds
~edEvt<Name>Count~~edEvtBountyCount~how many times the event occurred this session
~edEvt<Name>Last~~edEvtBountyLast~timestamp of the latest occurrence
~edEvt<Name>_<Field>~~edEvtBounty_TotalReward~each field of the latest occurrence

Fields that the newest occurrence doesn't have are removed automatically, so you never read a stale leftover from a differently-shaped earlier event.

Globals — companion files

Each companion file's top-level fields are flattened to ~edFile<Name>_<Field>~ — e.g. ~edFileCargo_Count~, ~edFileMarket_StationName~, ~edFileBackpack_Items~. Two convenience extras:

VariableHolds
~edCargoUsed~cargo units used (from Cargo.json)
~edNavRemainingJumps~ / ~edNavDestination~also refreshed from NavRoute.json whenever you plot a route

Recipe — do something when undocking in a specific system

  1. Create an action; right-click its Triggers → Custom → Elite Dangerous → Ship State → Docked Off.
  2. Add Core → Logic → If/Else: Input %edSystem% · Operation Equals · Value Shinrarta Dezhra.
  3. Streamer.bot adds True Result and False Result groups under the If/Else. Put your sub-actions (key press, sound, chat line…) inside True Result; leave False Result empty.
Why %edSystem% and not ~edSystem~ here?

Both actually work in this action — the flag trigger delivers %edSystem% as an argument, and ~edSystem~ reads the same value from the live global. Prefer the argument when the trigger provides one; reach for the tilde when it doesn't.

Recipe — a TTS line with live values

Trigger: Journal Events → FSDJump. Add a Speaker.bot → Speak (or any TTS) sub-action with the message:

Arrived in %edEvent_StarSystem% — jump ~edJumps~ of the session, ~edNavRemainingJumps~ to go.

Arguments and globals mix freely in one input — the argument holds the jump you just made, the tildes read your running totals.

Recipe — a !system chat command

Because globals work from any trigger, your Elite state is usable in ordinary Twitch commands:

  1. Create a Streamer.bot Command !system pointing at a new action.
  2. Add a Twitch → Chat → Send Message to Channel sub-action:
CMDR ~edCmdr~ is in ~edSystem~ flying a ~edShip~ — ~edJumps~ jumps and ~edCreditsEarned~ credits this session.

No Elite trigger involved — the watcher keeps those globals current in the background.

Recipe — low-fuel klaxon

  1. Create an action; trigger: Custom → Elite Dangerous → Ship State → LowFuel On.
  2. Add Core → Sounds → Play Sound with your klaxon file. Done — the game raises the flag at 25% main tank.

Want your own threshold instead? Trigger on Journal Events → FSDJump and If/Else on %edEvent_FuelLevel% < your number.

Fine print — flattening rules & limits

  • Nested objects flatten with underscores up to two levels deep (%edEvent_SystemFaction_Name%); anything deeper arrives as a JSON string.
  • Arrays publish a …Count value (e.g. %edEvent_RouteCount%) plus the array itself as JSON when it fits in 400 characters.
  • Field names are sanitized to letters, digits, and underscores — punctuation in a journal field name is dropped.
  • Numbers: floats are rounded to 4 decimal places; long strings are truncated at 400 characters; the full-event %edEventJson% is capped at 4,000.
  • Booleans arrive as True/False — with Auto Type on, If/Else compares them fine against True.
  • Variable cap: the watcher stops creating new variable names at 5,000 (existing ones keep updating); a warning appears in the log if that ever happens.
  • Localised fields: many events have both a raw and a _Localised field (e.g. Target and Target_Localised) — prefer the localised one for anything shown on stream.
