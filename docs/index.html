Elite Streambot — User Guide

Turn what happens in your Elite Dangerous session into live alerts on your stream — no coding required.

Introduction

Elite Streambot watches your Elite Dangerous game journal in real time and tells Streamer.bot to run an action whenever something happens — you collect a big bounty, get interdicted, make a first discovery, run low on fuel, and so on. Streamer.bot then does whatever you want with it: play a sound, speak a text-to-speech line, post in chat, flash an OBS overlay.

You set it up once, leave it running while you play, and your alerts fire automatically with the real details filled in — the reward amount, the system name, who killed you, your running session totals.

Who this is for

Elite Dangerous streamers who use Streamer.bot. You do not need to know how to program — the app has a visual rule builder and ships with ready-made examples.

How it works

┌─────────────────┐ ┌──────────────────────┐ ┌──────────────┐ ┌─────────────────┐ │ Elite Dangerous │ ───► │ Elite Streambot │ ───► │ Streamer.bot │ ───► │ sounds · TTS · │ │ journal files │ │ rules + templates │ WS │ your actions │ │ OBS · chat · … │ └─────────────────┘ └──────────────────────┘ └──────────────┘ └─────────────────┘

Elite Dangerous writes a running log (its "journal") of everything you do. Elite Streambot reads that log as it's written, checks each event against your rules, and when a rule matches it sends a message to Streamer.bot over a local connection asking it to run one of your actions — passing along useful values you can use in your alert.

Everything runs locally on your PC. Nothing is sent to the internet.

1Install & run

No installer and no other software required — the app is a single self-contained program.

  1. Download elite-streambot-win-x64.zip from the Releases page.
  2. Unzip it to a normal folder such as Documents. Avoid Program Files — the app saves your settings next to itself and needs write access.
  3. Double-click EliteStreambot.exe. A small console window opens and stays open while the app runs.
  4. Open http://localhost:8377 in your browser. This is your dashboard.
Windows SmartScreen warning

Windows may warn that the app is unsigned and from an unknown publisher. This is expected — code-signing certificates cost money and this is a free tool. Click More info → Run anyway. (You can inspect the full source on GitHub if you'd like to build it yourself instead.)

Keep the folder together

The exe reads its public/, rules/, and sounds/ folders from right beside itself, and writes your settings to config.json there. Move the whole folder as a unit, not just the exe.

2Connect Streamer.bot

Elite Streambot talks to Streamer.bot through Streamer.bot's built-in WebSocket server. You just need to switch it on.

  1. Open Streamer.bot.
  2. Go to the Servers/Clients tab → WebSocket Server.
  3. Click Start Server (and tick Auto Start so it comes up automatically next time). The defaults — address 127.0.0.1, port 8080 — match Elite Streambot's defaults, so there's nothing else to configure.

Back on the dashboard, the Streamer.bot badge in the top-right turns green and shows the detected version within a couple of seconds.

Different address or port?

If you changed Streamer.bot's WebSocket settings, click ⚙ Settings in the dashboard and enter the matching host / port / endpoint. Changes apply immediately.

3Create actions in Streamer.bot

Elite Streambot fires your Streamer.bot actions by name. So each rule needs a matching action to exist in Streamer.bot, or the alert has nothing to run.

The preset rules expect actions named ED Docked, ED Big Bounty, ED Died, and so on (see the full list). To create one:

  1. In Streamer.bot, open the Actions tab.
  2. Right-click in the actions list → Add, and name it exactly to match a rule, e.g. ED Big Bounty.
  3. Repeat for each alert you want. (An action with no sub-actions is valid — it just does nothing yet. That's fine for testing the connection.)
You don't have to use the preset names

Prefer to point a rule at an action you already have? Click Edit on any rule in the dashboard and change its action name. The rule and the action just have to agree.

The magic part: variables

Every value a rule sends arrives inside your action as a %variable%. For example the Big Bounty rule sends %reward% and %target%, so a Text-to-Speech sub-action reading Commander just bagged a %reward% bounty on a %target%! speaks the real numbers. Build the action once; Elite Streambot fills in the data every time.

4Add alert sounds

The download includes a sounds/ folder with a distinct cue for every preset alert (a coin arpeggio for bounties, a klaxon for low fuel, an ominous drone for Thargoids, and so on). To make an action play one:

  1. In Streamer.bot, click the action (e.g. ED Died).
  2. In the Sub-Actions panel, type play sound in the search box and choose Core → Sounds → Play Sound.
  3. Browse to the matching file in the app's sounds/ folder (e.g. died.wav).
  4. Save. Then hit the rule's Test button in the dashboard — you'll hear it immediately.

Swap in your own .wav or .mp3 files any time — nothing is locked to the bundled sounds. For spoken alerts, add a Text-to-Speech sub-action instead (this needs Streamer.bot's companion app, Speaker.bot).

Creating rules

A rule says: when this game event happens (and matches your conditions), run this Streamer.bot action (with these values). The easiest way to make one is the visual builder.

  1. Click + New Rule (or + rule on any live event to start from it, or Edit on an existing rule).
  2. When this game event happens — start typing an event name; the box suggests events it knows about, including ones from your own journal.
  3. Only if… — add optional conditions as simple rows (field · is / at least / contains… · value). Leave empty to fire every time.
  4. Run this Streamer.bot action — auto-completes from your connected Streamer.bot actions.
  5. Send these values — pick game values from a dropdown; each becomes a %variable% in your action.

As you go, the builder shows a plain-English summary and a preview of the rule that will be saved. Rules are stored as small text files in the rules/ folder and reload automatically when changed, so you can also edit them by hand if you prefer.

"Why isn't my rule firing?"

Check its counters. seen 0× means the trigger event name never arrived — check the spelling against the Live Events feed. seen 12× · fired 0× means the event is arriving but your condition is filtering it all out.

Global variables

Beyond firing alerts, Elite Streambot can publish your live Elite state into Streamer.bot itself as global variables — visible in Streamer.bot under (x) Global Variables → Persisted Globals, updating automatically as you play. Any action, command, or overlay you build in Streamer.bot can then read them, with Elite Streambot completely out of the loop.

ExamplesHolds
%edSystem% / %edStation%where you are right now
%edJumps% · %edCreditsEarned% · %edBounties% · %edDeaths%running session totals
%edLandingGearDown% · %edHardpointsDeployed% · %edSupercruise% · %edShieldsUp%live ship state (true/false)
%edLastEvent%name of the most recent journal event

That enables things like a !system chat command that answers with your current system, an OBS text source showing session credits, or actions that check ship state — all built entirely in Streamer.bot.

One-time setup

Streamer.bot has no API for writing globals from outside, so a tiny helper action inside Streamer.bot does the writing. You create it once; after that every variable (including ones added in future app updates) appears automatically.

  1. In the dashboard's Streamer.bot Variables panel, flip the toggle on and click Setup….
  2. Follow the five steps shown: create an action named ED Set Globals, add a Core → C# → Execute C# Code sub-action, paste the code from the Copy C# code button, hit Compile, then Save.
  3. Back in the dashboard, click Verify — it reads the variables back out of Streamer.bot and confirms the sync is live.
How it stays efficient

Only changed values are sent, batched roughly once a second — a busy session doesn't flood Streamer.bot, and a full refresh is one click (Publish now) or automatic on reconnect.

Standalone alternative — no app needed

Prefer everything inside Streamer.bot, including native triggers for every game event? See the Standalone C# watcher — a single paste-in action that does all of this (and more) with no external app running. Use it or the app's sync, not both.

Standalone C# watcher

extras/ed-journal-watcher.cs is the whole bridge condensed into one Streamer.bot C# action. It reads the Elite Dangerous journal itself on a background worker inside Streamer.bot — the Elite Streambot app doesn't need to be running (or even installed). It compiles with Streamer.bot's default references — no additions in the References tab needed. Paste it once and Streamer.bot itself understands Elite Dangerous.

Example — "when my landing gear comes up, press L"

Create an action → right-click its Triggers → Custom → Elite Dangerous → Ship State → LandingGearDown Off → add whatever sub-actions you like (a key press, a sound, a chat line). That's the entire setup. All Ship State triggers (every flag, On and Off) are in the menu from the moment the watcher starts; the per-event-type triggers under Journal Events appear once that event has been seen at least once and are remembered across Streamer.bot restarts.

Variables, same as the app's sync — and then some

  • Everything described under Global variables: session aggregates, complete Status.json state (every flag by name, pips, GUI focus, fuel, position, …).
  • A mirror of every journal event: %edEvt<Name>Count%, %edEvt<Name>Last%, and each field of the latest occurrence as %edEvt<Name>_<Field>% (e.g. %edEvtDocked_StationName%).
  • Companion files flattened to variables (%edFileCargo_Count%, …) plus navigation state: %edNavTargetSystem%, %edNavRemainingJumps%, %edNavDestination%.

Built to be left alone

  • Checkpoint/resume — it remembers its exact position in the journal, so restarting Streamer.bot never replays old events as triggers and never loses new ones.
  • First run is quiet — it rebuilds state from your current journal without firing a trigger for every historical line, and a big offline backlog updates variables while suppressing the trigger flood.
  • Steady + safe — it checks the journal once a second (negligible load); journal rollovers and mid-write files are handled.

Setup (once)

  1. In Streamer.bot: Actions → right-click → Add → name it ED Journal Watcher.
  2. Sub-Actions → right-click → Core → C# → Execute C# Code.
  3. Paste the entire contents of ed-journal-watcher.cs and click Compile (it should report OK).
  4. In the sub-action's settings, tick “Precompile on Application Start” — that makes the watcher register its triggers and start automatically whenever Streamer.bot starts.
  5. Save, then right-click the action → Test Action to start it right now. Variables appear under (x) Global Variables → Persisted Globals.

App sync or standalone — which one?

Pick the app's variable sync if…Pick the standalone watcher if…
you want the dashboard: alert rules with conditions and cooldowns, the visual builder, the event simulator, live diagnostics — with variable sync as one feature among many. you want zero extra software running, native Elite triggers in Streamer.bot's own menu, companion-file data, and variables that keep flowing even when the app is closed.
Don't run both syncs

Both write the same ed* variables. Running the app's variable sync and the standalone watcher at the same time just writes everything twice. (The app's alert rules are fine to use alongside the watcher — it's only the two variable syncs that overlap.)

Rule reference

Here is a complete rule as it's stored on disk. Every field except name, trigger, and action is optional.

name: Big Bounty
enabled: true
trigger: Bounty                      # event name, a list, or "*"
when: event.TotalReward >= 250000    # optional condition
cooldown: 20                         # min seconds between firings
action: ED Big Bounty                # Streamer.bot action to run
args:                                # each becomes a %variable%
  reward: "{{event.TotalReward | credits}}"
  target: "{{event.Target_Localised}}"

Triggers

Any journal event name (FSDJump, Docked, Bounty, Died, Scan, …), a list of names, or "*" for every event. The app also creates its own events from the game's status file when your ship's state changes:

Status.LowFuel Status.Overheating Status.InDanger Status.BeingInterdicted Status.ShieldsDown Status.ShieldsRestored

Conditions (when)

Conditions are checked safely — a rule file you got from someone else can't run harmful code. You can use:

  • Values: event.*, status.*, session.*, numbers, 'text', true/false
  • Comparisons: === !== >= <= > <
  • Logic & math: && || ! ( ) and + - * / %
  • Functions: contains, startsWith, endsWith, lower, upper, len, abs, round, min, max

Examples: event.TotalReward >= 250000 · session.jumps % 10 === 0 · contains(event.StarSystem, 'Colonia')

Stopping the app

Any of these stops Elite Streambot cleanly:

  • Click ⏻ Quit in the top-right of the dashboard (recommended) — it confirms, shuts the server down, and shows a "stopped" screen.
  • Press Ctrl + C in the app's console window.
  • Close the console window.

While the app is stopped, no alerts fire. Start EliteStreambot.exe again and reopen the dashboard to resume.

FAQ

Do I need to keep the browser open?

No. The dashboard is just a control panel. Alerts keep firing as long as the exe (console window) is running, even with the browser closed.

Does this send my data anywhere?

No. Everything runs on your PC. The app only connects to Streamer.bot on your own machine, and the dashboard is reachable only from your computer (localhost).

Do I need Node.js or anything else installed?

No. The Windows download bundles everything into the single EliteStreambot.exe.

Can I test alerts without playing?

Yes — use the Simulator panel or any rule's Test button. Both run the full pipeline through to Streamer.bot.

Will restarting the app reset my session totals?

No. On startup it re-reads your current journal to rebuild the session stats. Replayed events update the totals but never fire alerts.

Is it safe to leave running the whole stream?

Yes. It's lightweight — a fraction of a percent of one CPU core at idle and a few dozen MB of memory.
